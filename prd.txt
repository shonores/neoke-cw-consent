0.1 What Is a Digital Credential Wallet?
A digital credential wallet is a web application that allows a person or organisation to:
1.​ Receive signed digital documents from issuers (universities, governments, employers)
— called credentials. Examples: a driving licence, an employee badge, an age
verification proof.
2.​ Store those credentials securely.
3.​ Present selected credentials (or selected fields from them) to verifiers (border controls,
shops, partner services) — called a presentation.
Credentials are cryptographically signed. The holder cannot forge them, and any verifier can
check the issuer's signature without contacting the issuer. Think of it as a tamper-proof digital
passport.

The wallet in this project is a web application — a React SPA running in a browser. It connects
to a Neoke IDN Node (a cloud-hosted identity server) via an HTTP API. The node holds the
actual cryptographic keys and signed credentials. The wallet is the UI layer that lets a human
trigger and observe operations on that node.

0.3 What Are the Credential Formats?
Two formats are in use:
mso_mdoc (Mobile Document / ISO 18013-5): Used for government-style identity documents
(driving licences, Photo IDs). Claims are organised into namespaces — a two-level map like:
json
{ "org.iso.18013.5.1": { "family_name": "Smith", "portrait": "..." } }

dc+sd-jwt (Selective Disclosure JWT): Used for privacy-sensitive credentials. Each field can
be independently hidden or revealed. Claims are a flat JSON object:
json
{ "over_18": true, "nationality": "NL" }

Both formats are received via the same issuance flow and presented via the same verification
flow. The wallet handles both transparently.

0.4 What Are Credential Offer URIs and VP Request URIs?
Credential offer URI (issuance): A special deep link an issuer sends to invite a wallet to receive
a credential.
text
openid-credential-offer://?credential_offer_uri=https%3A%2F%2F...

VP request URI (verification): A special deep link a verifier sends to request credential
presentation.
text
openid4vp://?client_id=did%3Aweb%3A...&request_uri=https%3A%2F%2F...

Both arrive as URL query parameters when the user follows a link:
text
https://wallet.example.com/?uri=openid4vp%3A%2F%2F%3F...

The wallet's existing detectUriType() classifies any raw string as receive, present, or
unknown. The classification determines which screen handles the link.

0.5 How Does the Existing Wallet Work Today?
The existing wallet is built with React 18, TypeScript, Tailwind CSS, and Vite. It has no routing
library — navigation is a simple currentView state variable in App.tsx with a navigate()
function. There is no URL-based routing.
Authentication flow:
1.​ Onboarding Step 1: User enters a node identifier (e.g., b2b-poc). The wallet validates
the node is reachable.
2.​ Onboarding Step 2: User enters their API key. The wallet calls POST /:/auth/authn,
receives a bearer token, and stores it in localStorage with an expiry timestamp.
3.​ AuthContext manages token state, expiry monitoring, and session restore on page
reload.
Main screens:
●​ DashboardScreen — Shows the credential card carousel and navigates to credential
detail.
●​ CredentialDetailScreen — Shows full fields of a selected credential.
●​ ReceiveScreen — Handles credential offer URIs end-to-end (shows offer details, calls
/:/oid4vci/receive).

●​ PresentScreen — Handles VP request URIs end-to-end (calls preview, shows what
will be shared, calls respond on user approval).
●​ AccountScreen — Shows current node connection, logout.
●​ OnboardingStep1Screen / OnboardingStep2Screen — Two-step login flow.
●​ ReAuthModal — Overlay shown when token expires mid-session.
Bottom tab bar: Three tabs — Home (dashboard), Scan QR Code (navigates to receive),
Account.
Deep link routing: On mount, App.tsx reads ?uri= or ?offer_uri= from
window.location.search, classifies the URI, and after authentication, navigates directly to
ReceiveScreen or PresentScreen with the URI as initialUri.
State storage: localStorage keys:
●​ neoke_s_token — bearer token
●​ neoke_s_expires — token expiry (ms timestamp)
●​ neoke_node_id — node identifier (short form)
●​ neoke_activity — last activity timestamp (for 7-day inactivity lock)
●​ neoke_credentials — locally cached credential list (merged server/local)
Existing API client: src/api/client.ts exposes all node API calls. The module-level
_baseUrl is set by setBaseUrl() during onboarding and session restore. Every API function
takes a token string.

0.6 What Is the Consent Engine (Layer 2) and Why Does
the Wallet Need to Know About It?
The Consent Engine is a separate HTTP microservice (a Node.js/TypeScript process running
at its own URL, e.g., https://consent.example.com). It was built as Layer 2 and sits
between the wallet and the IDN Node for automated, policy-driven decisions.
What it does (relevant to the wallet):
1.​ Intake: It accepts any credential-related URI (VP request, credential offer, verification
link) and evaluates it against a database of user-defined rules.
2.​ Automated decisions: If a matching rule exists, it acts automatically:
○​ For VP requests: calls the IDN Node's preview → respond on the user's behalf.
○​ For credential offers: calls the IDN Node's receive on the user's behalf.
○​ Returns action: "auto_executed" immediately.

3.​ Queue: If no rule matches, it puts the request into a pending queue and returns
action: "queued". The user must later approve or reject it.
4.​ Rejection: If a request cannot be handled (no credentials match, offer expired, etc.), it
returns action: "rejected".
5.​ Rules management: It exposes a full CRUD API for managing the rules that drive its
decisions.
6.​ Audit log: It records every decision (auto or manual) with full context.
Why the wallet needs to integrate:
●​ Without Layer 3, the wallet is unaware of the Consent Engine. Users still process every
link manually (scan → see → approve → done). This is fine for interactive use but
misses the automation benefit.
●​ With Layer 3, the wallet becomes the management UI for the Consent Engine: users
create rules, review queued requests, see audit history, and observe automated
decisions — all from the same wallet they already use.
●​ Additionally, deep links that arrive in the wallet should be routed through the CE (when
CE is configured) instead of processed directly, so that rules are enforced even for links
the user opens manually.
Key principle: The Consent Engine is optional. If the user has not configured a CE URL, the
wallet behaves exactly as it does today — direct mode, no changes to existing flows. Every new
CE-specific feature is additive and non-breaking.

0.7 How Does the Consent Engine API Work?
The CE exposes a REST API at its configured base URL. Authentication uses the same API key
mechanism as the IDN Node:
text
Authorization: ApiKey {api-key}

The CE validates this key against its configured IDN Node. The same API key the user entered
at wallet onboarding (for the IDN Node) is also valid for the CE, because the CE authenticates
by verifying the key against the node it is bound to. No separate CE password is needed.
Core intake endpoint:
text
POST {ceBaseUrl}/consent/intake
Authorization: ApiKey {api-key}

Content-Type: application/json
{
"rawLink": "openid4vp://?client_id=...&request_uri=..."
}

Response:
json
{
"requestId": "uuid",
"action": "auto_executed",
"linkType": "vp_request",
"reason": null
}

Possible action values:
●​ "auto_executed" — a matching rule was found and the operation was performed
automatically.
●​ "queued" — no rule matched; the request is now in the pending queue.
●​ "rejected" — the request was rejected (no matching credentials, expired link, etc.).
●​ "manual_required" — the CE was unable to process this link type; the wallet should
handle it directly using the existing flow. This is the fallback for edge cases the CE
cannot handle.
Rules API (abbreviated):
●​ GET /rules — list all rules
●​ POST /rules — create a rule
●​ GET /rules/{id} — get a rule
●​ PATCH /rules/{id} — update a rule
●​ DELETE /rules/{id} — delete a rule
●​ POST /rules/{id}/enable — enable a rule
●​ POST /rules/{id}/disable — disable a rule
●​ POST /rules/{id}/test — dry-run: returns wouldMatch: true/false for a
given link without executing
Queue API:

●​ GET /queue — list pending requests (filter by status: "pending")
●​ GET /queue/{id} — get one pending request with full context (verifier info, requested
fields, etc.)
●​ POST /queue/{id}/approve — approve and execute
●​ POST /queue/{id}/reject — reject without executing
●​ DELETE /queue/{id} — delete from queue
Audit API:
●​ GET /audit — list audit events with optional filtering (?nodeId=, ?action=,
?limit=, ?offset=)
Discovery API (for rule creation wizard):
●​ GET /node/{nodeId}/credentials — list credential types and their available
claims from the node
Health:
●​ GET /health — returns { status: "ok" | "degraded", nodeConnected:
boolean, pendingCount: number }

The CE URL is stored in localStorage separately from AuthContext. It is scoped to the

previous session should be cleared. To enforce this, clear neoke_ce_url and
neoke_ce_enabled in AuthContext's logout() handler.

3.2 CE Configuration Rules
●​ CE is always optional. If neoke_ce_url is absent or empty, the wallet behaves
exactly as today.
●​ The CE URL must be a valid https:// URL. Validate with a simple check: starts with
https:// and contains at least one . after the scheme.
●​ The CE API key is the same API key the user entered at onboarding Step 2. It is not
stored separately in localStorage (it is kept only in memory as part of the auth flow).
To authenticate with the CE, the wallet uses Authorization: ApiKey {apiKey} —
but this requires the raw API key. Problem: the wallet does not retain the raw API key
after onboarding (it only stores the resulting bearer token).
Solution: When the user configures the CE URL (in AccountScreen), ask them to enter their
API key once more. Store it encrypted in localStorage under neoke_ce_apikey using the
same simple approach used for session tokens (no deep encryption is needed for MVP —
localStorage is the boundary). This is consistent with the Layer-2 PRD's note that API keys can
be treated as plaintext in controlled environments.
Alternatively (simpler for MVP): add a step to onboarding Step 2 where, if a CE URL has
already been configured from a previous session, the CE API key is refreshed automatically by
re-using the key entered in the current session.
Recommended implementation: Store the raw API key in localStorage under
neoke_ce_apikey at the time the user completes onboarding Step 2. Clear it on logout. This
makes the key available whenever the CE needs to be called, without re-prompting the user.

6.3 Behaviour
On mount / context initialisation:
1.​ Read neoke_ce_url, neoke_ce_enabled, neoke_ce_apikey from
localStorage.
2.​ If CE URL exists and enabled: call setCeBaseUrl() on the CE client module and call
refreshHealth().
3.​ Set initial isConnected and pendingCount from health response.
configureCe(ceUrl, apiKey):
1.​ Validate ceUrl reachability via validateCeUrl().
2.​ Validate apiKey by calling checkCeHealth() with it — if the health endpoint returns
nodeConnected: false, warn but do not block.
3.​ Store ceUrl, apiKey, and enabled: true in localStorage.
4.​ Update context state.
5.​ Call setCeBaseUrl(ceUrl).
refreshHealth():
1.​ Call checkCeHealth().
2.​ Update isConnected, pendingCount, lastChecked.
Background polling:
●​ Poll refreshPendingCount() every 30 seconds when CE is configured and enabled.
●​ Use setInterval in a useEffect with the standard cleanup pattern.
●​ Do not poll when ceEnabled is false.

On logout (triggered by AuthContext):
●​ When useAuth().state.token becomes null, reset CE context state to defaults but
do not clear localStorage CE config — the user may be re-authenticating to the
same node.
●​ Clear CE config (localStorage + state) only when useAuth().logout() is called
(full sign-out, not token expiry).

SECTION 12 — CE CLIENT INITIALISATION AND
LIFECYCLE
12.1 Startup Sequence
On app mount:
text
1. AuthProvider initialises
└── Reads neoke_s_token, neoke_s_expires, neoke_node_id from
localStorage
└── If token present and not expired: restores session (sets
baseUrl, token)
└── If token expired: sets sessionExpired flag (ReAuthModal
appears)
└── If no token: shows onboarding
2. ConsentEngineProvider initialises (after AuthProvider)
└── Reads neoke_ce_url, neoke_ce_enabled, neoke_ce_apikey from
localStorage
└── If CE URL present and enabled:
└── Calls setCeBaseUrl(ceUrl)
└── Calls refreshHealth()
└── Updates isConnected, pendingCount
└── Starts background poll (every 30 seconds)
└── Calls refreshPendingCount() → GET /health
└── Updates pendingCount in state (triggers badge re-render)

12.2 Token vs API Key

The IDN Node uses a bearer token for all calls (short-lived, ~1 hour). The CE uses an API key
for all calls (the same raw API key entered at onboarding, stored in localStorage under
neoke_ce_apikey).
This is intentional: the CE is a separate service and does not share the node's token lifecycle.
The API key does not expire. The CE validates it by making its own auth call to the node.
Implication for the wallet:
●​ When calling any CE API, always pass the apiKey from ConsentEngineContext
state.
●​ If ceApiKey is null (not stored), CE calls cannot be made. Show a "reconnect" prompt in
AccountScreen.
●​ Never pass the bearer token (from AuthContext) to the CE. These are different
credentials for different services.

12.3 Handling CE Errors
text
CE unreachable (network error)
└── Update isConnected: false in context
└── Show CeStatusBanner "⚠ Consent Engine offline"
└── Fall back to direct mode for any new link
CE returns 401
└── API key is invalid or revoked
└── Show AccountScreen prompt: "Consent Engine API key rejected —
reconnect"
└── Clear neoke_ce_apikey from localStorage
└── Set isConnected: false
CE returns 503 (degraded)
└── isConnected: false
└── Log warning, continue with degraded banner
└── Do not fall back to direct mode for already-queued items — they
stay in queue

SECTION 21 — OPEN QUESTIONS FOR PRODUCT
REVIEW
Document these as TODO comments in the code:
1.​ Scan tab behaviour with CE: Currently tapping "Scan QR Code" navigates to
ReceiveScreen. With CE routing added, should the tab still be labelled "Scan QR
Code", or should it be relabelled something more neutral like "Scan" since the user no
longer directly controls what happens after scanning?
2.​ CE API key rotation: If the user rotates their node API key (new key issued, old one
revoked), they must update the CE's stored key in AccountScreen. There is no
automatic rotation notification. Should AccountScreen show the last-successful-auth
timestamp to help users detect stale keys?
3.​ Queue item expiry display: The CE's default TTL for queued requests is 24 hours. The
wallet should show countdown timers ("Expires in 23h 45m"). At what threshold should
the "Expiring soon" warning appear? Current proposal: 1 hour. Confirm.
4.​ Approval of credential offers that require a PIN: When preview.requiresPin:
true, the wallet must ask the user for the PIN before calling approveQueueItem().
The CE will pass it to the node. However, if the issuer's PIN has expired, the CE will

return an error. How should the wallet communicate this to the user? Proposal: show
"This offer's PIN may have expired — contact the issuer for a new one."
5.​ Multiple credentials matching one VP request: When the CE previews a VP request
and matchedCredentials contains more than one entry for the same credential type
(e.g., two driving licences), the CE picks the most recently issued one. The wallet
currently has no way to indicate which credential was used. Should the audit log entry
include the credential ID that was selected? Confirm with CE team.
6.​ Offline / backgrounded wallet: The CE processes requests autonomously on the
server even when the wallet is closed. When the user next opens the wallet,
refreshHealth() fires and pendingCount is updated. Is the 30-second background
poll sufficient for the expected latency requirements, or does the use case require push
notifications (future scope)?
